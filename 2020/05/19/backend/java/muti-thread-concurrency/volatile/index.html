volatile的特性

保证线程可见性

加了volatile在字节码方面，多了一个ACC_volatile的标识

volatile基本数据类型

会发现不加volatile的时候，程序会一直运行，但是加了volatile之后程序停止运行

当然自己在测试期间走了个弯路，自己喜欢加System.out.println方法打印出实时运行的情况，于是自己刚开始是这么写的

发现是验证不了线程不可见性这种情况，但是去掉System.out.println("m start");就能验证.点进println方法，发现此方法加了锁

于是查资料得到
JMM关于synchronized的两条规定:

  1. 线程解锁前，必须把共享变量的最新值刷新到主内存中
  2. 线程加锁时，将清空工作内存中共享变量的值，从而使用共享变量时需要从主内存中重新获取最新的值

synchronized具体过程是：

  • 获得同步锁；
  • 清空工作内存；
  • 从主内存拷贝对象副本到工作内存；
  • 执行代码(计算或者输出等)；
  • 刷新主内存数据；
  • 释放同步锁。

Thread.sleep ，println 都会影响到 《《当前》》 线程 将副本与主存同步

引用类型（包括数组）

只能保证引用本身的可见性，不能保证内部字段的可见性

发现线程并不会停止，但是改成如下,将会停止

volatile并不能保证多个线程共同修改running变量时所带来的不一致问题，也就是说volatile不能替代synchronized

只能做到不同时执行的线程的可见性

禁止指令重排序

Java 里只有 volatile 变量是能实现禁止指令重排的
volatile 是在底层通过内存屏障防止指令重排的，变量前后之间的指令与指令之间有序可见

不能保证原子性
