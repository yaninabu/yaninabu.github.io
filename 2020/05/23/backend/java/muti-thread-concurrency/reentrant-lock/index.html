可重入锁

ReentrantLock可以代替synchronize

但是需要lock.lock()/lock.unlock()
写到try {} final {lock.unlock()}, 必须写到final里面，因为有无异常都会执行
使用syn锁定的话如果遇到异常，jvm会自动释放锁，但是lock必须手动释放锁，因此经常在finally中进行锁的释放

为啥什么有了synchronize还要有ReentrantLock？

  1. ReentrantLock里面有封装了比synchronize好用的方法
    • tryLock() 尝试拿，在规定时间里那不到就可以做自己想做的逻辑，也就是等待或处理都可以；但是synchronize只能是等待
    • lockInterruptibly可以对打断的加锁
    • 公平锁
      • 如果有N个线程在队列里面等待，如果不是公平锁N+1个线程来了直接抢锁；但是公平锁，直接去队列里查看有无等待的线程，如果有自己也等待
