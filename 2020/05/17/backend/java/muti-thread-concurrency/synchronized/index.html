synchronized的特性

1. 对对象上锁

上锁对象可以任意指定，利用对象上面的两位值，去判断是否能拿到锁

在static方法上面加了static，相当于锁了T.class。要注意的是不能锁String/Integer/Long等基本数据类型的引用

2. synchronized 允许加锁和不加锁的方法同时执行

如果test()方法也加上synchronize，就变成了一个运行完另一个再运行
加完锁之后效率低一百倍，如果业务允许脏读，读操作能不锁就不要加锁

3. 可重入锁

必须是可重入锁，不然会造成死锁现象
比如两个synchronize的方法a，b；a中调b
如果是不可重入调b方法的时候发现this上面加了一把锁，等待a释放，a又等待b执行
也就是一个线程已经拥有某个对象的锁，再次申请的时候仍然会得到该对象的锁

或者是子类调父类里面的方法问题

4. 程序之中遇到异常，默认情况下锁会被释放

如果A线程在执行，B.C线程在Block，如果a抛出异常，B.C马上就有可以拿到锁的机会，这样会造成数据不一致的情况。

如果不想释放锁，想要继续循环，可以加try、catch

synchronize的底层实现

  1. JDK早期1.6之前是去操作（内核）去申请锁。重量级锁。
  2. JDK高版本1.6z之后包含1.6改进成了锁升级的方式
    • 第一个线程到来先加偏向锁（乐观锁）。所谓偏向锁就是对加锁对象头的的mark word用两位来记录锁的信息。
    • 第二个线程到来，升级成自旋锁（默认自旋10次）。占CPU在用户态，不占内核态效率高。
    • 自旋10次之后，如果还没抢占到资源，升级为重量级锁（去OS中申请）。不占CPU，在等待状态，只有CPU叫你你才起来，占内核态效率低

CAS/重量级锁的使用场景

  1. 执行时间短，线程数比较少的时候用自旋锁
  2. 执行时间长，线程数比较多的时候用重量级锁

synchronized的优化

  1. 同步锁的细化
    • 如果有业务上的逻辑在一个方法上，就只对该该锁的值得范围上加锁
  2. 锁的粗化
    • 如果锁定的值比较多，就不要多次加锁，一次性加一个锁
  3. 对锁定对象加 final修饰 final private Object o = new Object()
    • 原因是如果对象的属性发生了改变，不影响锁的使用，但是如果对象的引用发生了改变，就会影响
  4. 不要用String/Integer等基本数据类型的类作为加锁对象
