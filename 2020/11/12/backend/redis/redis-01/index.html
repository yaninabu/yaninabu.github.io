数据的发展过程

  1. 刚开始数据存放在文件中，放到磁盘上，磁盘寻址是毫秒级的，一个扇区是512b，但读取起来很慢，并且以4K为单位读取。

  2. 出现了数据库，出现了表，一块一块默认按照4K存储，也是在磁盘上，如果没有索引，速度并没有提高，和上面的发展过程是一样的。

  3. 创建索引，索引的叶子节点是在磁盘上面的，但是B+树的树干（区间和偏移）是在内存里面的，内存的寻址是纳秒级别是磁盘的10W倍，先从B+树的树干中查找到叶子节点，然后将叶子节点从磁盘中读到内存，循环遍历符合条件的数据的引用，在通过引用去找磁盘中符合条件的数据

为什么redis发展快

redis是单线程（数据处理上单线程，真正的处理其他的事情的不是单线程）调用，但是为什么每秒处理几十万的请求？
内存是寻址是纳秒级别，而网络网卡是毫秒级别，差着10万倍，除非是10万个请求同一时间，到达，才能有秒级的响应，其他时候是纳秒毫秒级。
redis用了是基于内存的用了epoll技术，所有的client端连接先到系统的内核，然后才能到用户去处理，然而从内核到用户态的过程中用到了epoll。用户态端是基于内存的，是快的，而IO是慢的，client端慢慢悠悠传到内核，因为有epoll放到了map区，然后用户态端直接拿出来处理。那什么是epoll呢？

epoll的发展历程

  1. 刚开始socket是block的，也就是用户态和内核态没有数据返回就一直等待这是，也就是BIO
  2. 用户态开始非阻塞了，也就是一直轮询，切换查看文件描述符的数据fd，实现了NIO，但是造成的问题是查看每个文件描述符的数据状态，每次都会有用户态到内核态之间切换的问题，效率低。
  3. 内核态进行升级，将用户态的轮询放到了内核态，也就是select，以前轮询查看单个的文件描述符fd，但是现在我是把1000多个fds传给内核，然后内核区间空这个内容通过select()函数，知道一个或者更多变成了ready状态，就给用户态返回，然后用户拿着返回的文件描述符，也就是谁有了我就去调系统调用read(fd)。和以前先相比在系统调用上面更精准更精确了，减少了用户态和内核态不必的切换。实现了多路复用NIO
    但是每次我把所有的文件描述符传给你，然后你告诉我谁好了，然后我再去读。如果这个过程再精简就更好了。
    问题是用户态和内核态通过系统调用的函数，就有传参的问题，就会有数据的拷贝，于是乎大家都想做到零拷贝，所以问题就是拷贝来拷贝去的数据

    select做到了在
  4. 内核实现：mmap()的出现做一个映射，可以解决一个内核和用户组件一个共享的空间，也就是传参的时候不在传递数据的拷贝，而是传递一个地址，然后从共享空间里面拿东西。通过epoll的create方法产生一个epoll的文件描述符，以后每个连接进来之后，将这连接交给epoll的文件描述符，然后epoll这边会准备一个共享空间叫mmap，往共享空间的红黑树里面存放叫做注册，注意共享空间的增删改是内核来完成的，查看两方共同的，以前是1000个文件描述符来了通过传递给select函数，现在是通过epoll的create注册到红黑树里面，通过所有的IO中断，发现谁的数据到达了，谁的数据缓冲区有了，将到达的放到链表里，通过epoll事件通知设备，如果链表里面有数据了，用户态就可以从链表里面拿数据。
    用户空间还会去调用系统调用，调到epoll上，一个ctl的调用add或者delete真正的socket描述符，一个是wait(),等待，等待事件链表有了wait就返回

零拷贝和共享空间不一样

零拷贝叫sendfile(),一个输入的fd，一个输出的fd。
以前是文件file.tex先把fd到内核，然后read拷贝到用户空间，然后用户空间再Write copy到内核write出去到网卡，会有copy的过程。
有了sendfile(),直接调了sendfile，然后内核直接读缓冲区里面的数据，然后发出去，就没了拷贝

sendfile+mmap就出来了kafka

通过mmap写入挂载文件，然后这时候用户个内核共享，内核直接能看到这个数据，然后通过sendfile展示给客户。
一切的知识来自于内核

和nginx也很像

怎么启动nginx，要满足有多少个内核，就启动多少个worker（一个worker就可以把他的数据压到一二三级缓存了）有几个cpu就有几个nginx进程，每个进程使用的是多路复用epoll，同步非阻塞下的多路复用
